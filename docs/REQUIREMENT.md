# 요구사항 (onharu-backend-v2)

본 문서는 **현재 저장소 소스 코드**(주로 `interfaces.api.controller.I*`·구현체·`interfaces.shceduler`·`event`)를 근거로 정리한 **기능·비기능 요구사항**이다. 이전에 다른 프로젝트(영화 예매 등)용으로 작성된 내용은 **본 시스템과 무관**하므로 폐기한다.

---

## 1. 프로젝트 개요

| 항목 | 내용 |
|------|------|
| **시스템** | OnHaru 백엔드 API (`com.backend.onharu`) |
| **목적** | **결식 아동**과 **가게(사업자)** 를 연결하고, **가게 스케줄 기반 예약**, **이용 후 감사 리뷰**, **채팅·알림** 등을 지원하는 서비스의 서버 측 기능을 제공한다. |
| **형태** | REST API (`/api/...`) + **WebSocket(STOMP)** 채팅 메시지 전송, JSP 등 서버 렌더링 뷰 없음. |

---

## 2. 이해관계자·역할

| 역할 | 코드 상 표현 | 주요 책임 |
|------|----------------|-----------|
| **결식 아동** | `Children` API, 아동 회원가입·프로필 | 가게 검색·예약·리뷰·찜·(카드·증명서 등) |
| **사업자** | `Owner` API, 사업자 회원가입·프로필 | 가게 등록·수정·스케줄·예약 관리(확정·완료·취소) |
| **일반 사용자 계정** | `User` API | 로그인/로그아웃, 로컬·소셜 회원가입 흐름, 프로필, 탈퇴 |
| **관리자** | `Admin` API | 사업자·아동 **회원가입 요청 승인/거절** |
| **공통** | `Auth`, `Store`, `Chat`, `Notification` 등 | 인증 보조, 가게 조회, 채팅, 알림 |

---

## 3. 핵심 비즈니스 시나리오 (요약)

1. **회원**: 아동·사업자 각각 회원가입(증빙·사업자번호 등). 소셜 가입은 OAuth 이후 **마무리 API**로 프로필을 완성한다.
2. **관리자 검증**: 관리자가 가입 요청을 승인하거나 거절한다.
3. **가게**: 사업자가 가게를 개설하고 영업시간·태그·이미지 등을 등록한다. **엑셀 일괄 등록**으로 여러 가게를 넣을 수 있다.
4. **스케줄·예약**: 사업자가 날짜·시간대별 **예약 가능 스케줄**을 생성·수정·삭제한다. 아동이 스케줄을 선택해 **예약**하고, 필요 시 **취소**한다. 사업자는 예약을 **확정(WAITING→CONFIRMED)·완료·취소**한다.
5. **만료 처리**: 스케줄러가 주기적으로 **기한이 지난 예약**을 만료 처리한다.
6. **이용 후**: 아동이 예약과 연계된 **감사 리뷰**를 작성하고, 가게·전체 목록에서 조회한다.
7. **부가**: 가게 **찜**, **등급(Level)**, **알림 설정·히스토리**, **채팅방·메시지(WebSocket)**, **가게 검색 최근 검색어(Redis)** 등.

## 4. 기능 요구사항 (모듈별)

### 4.1 인증·계정 보조 (`Auth` — `/api/auth`)

- **사업자등록번호 진위**: 국세청 API 연동으로 사업자번호 확인 (`IAuthController` 설명).
- **아이디 찾기**, **비밀번호 재설정** 요청.
- **이메일 인증 코드** 발송·검증.
- **비밀번호 변경**, **현재 비밀번호 검증**.
- **SMS 인증** 관련 메서드는 소스에서 **주석 처리**되어 있으며, 현재 제품 요구사항으로 확정할 수 없다(미구현/보류).

### 4.2 사용자 (`User` — `/api/users`)

- **로컬 회원가입**: 아동·사업자 각각.
- **소셜 회원가입 마무리**: 아동·사업자 각각 (`finish` API).
- **로그인·로그아웃**, **세션 기반** `me` 조회.
- **프로필 조회·수정**(아동/사업자).
- **회원 탈퇴**(소프트 삭제 등 도메인 규칙은 구현 참고).
- **채팅 상대 선택용** 로그인 ID 부분 검색.

### 4.3 관리자 (`Admin` — `/api/admins`)

- 사업자 회원가입 요청 **승인·거절**.
- 결식 아동 회원가입 요청 **승인·거절**.

### 4.4 가게 (`Store` — `/api/stores`)

- 가게 **상세**·**검색(페이징·정렬 등)**.
- 가게 **스케줄 조회**: 월 단위 요약 또는 특정 일자 **시간대 슬롯** (`StoreSchedule` 연동).
- **카테고리** 목록.
- **엑셀 파일**로 가게 정보 **일괄 등록** (`StoreExcelFacade` / POI).

### 4.5 결식 아동 (`Children` — `/api/childrens`)

- **가게 예약 생성·취소**, **내 예약 목록·상세·요약**.
- 카드·증명서 등 API는 Swagger에서 `@Hidden` 으로 숨겨져 있으나 인터페이스상 **등록·수정·삭제·조회·재발급** 요구가 정의되어 있다.

### 4.6 사업자 (`Owner` — `/api/owners`)

- **가게 개설·수정·삭제(폐쇄)**, **내 가게 목록**.
- **예약 관리**: 목록(필터·페이징)·상세, **확정·완료·취소**.
- **예약 가능 스케줄** 생성·수정·삭제, **예약 요약** 조회.
- 일부 사업자 CRUD는 `@Hidden` 처리되어 있다.

### 4.7 감사 리뷰 (`Review` — `/api/reviews`)

- 예약과 연계된 **리뷰 작성**, **목록·가게별·내 리뷰** 조회 등.

### 4.8 찜 (`Favorite` — `/api/favorites`)

- 가게 **찜 토글**, **내 찜 목록** 조회.

### 4.9 등급 (`Level` — `/api/levels`)

- **등급** 생성·조회·수정·삭제 등(예약 완료 횟수 등 조건과 연계하는 도메인 모델 존재).

### 4.10 알림 (`Notification` — `/api/notifications`)

- 알림 **설정 조회·수정**, **히스토리 페이징**, 단건·전체 **읽음 처리**.

### 4.11 채팅 (`Chat` — `/api/chats`)

- 채팅방 **생성·수정·초대·목록·메시지 커서 조회·읽음·입장·퇴장** 등 REST 로 제공.
- **실시간 메시지 전송**은 `WebSocketConfiguration` 의 STOMP 엔드포인트(`/ws-chat` 등)와 `infra.websocket.ChatController` 의 `@MessageMapping` 으로 처리한다.

### 4.12 파일 업로드 (`S3` — `/api/upload`)

- **Presigned URL** 발급·파일 삭제 등(주석: 운영은 AWS S3, 개발/테스트는 MinIO 등 프로파일별).

### 4.13 가게 검색 최근 검색어 (`Store recent search` — `/api/stores` 하위 구현)

- 로그인 사용자 또는 **쿠키 `clientId`** 기반으로 **Redis** 에 최근 검색어 저장·조회.

## 5. 배치·도메인 이벤트

| 구분 | 내용 | 근거 |
|------|------|------|
| **예약 만료** | 매일 자정(`0 0 0 * * *`) `expireOverDueReservations` | `ReservationScheduler` |

| **예약 알림** | 예약 관련 이벤트 후 알림·히스토리 등 처리 | `event` + `ReservationEventListner` 등 |

## 6. 비기능 요구사항

### 6.1 보안·인증

- **Spring Security** + **OAuth2 Client** + **세션**(로그인 유지). 코드베이스 기준 **JWT 기반 API 인증은 사용하지 않는다**.
- CORS·세션 쿠키 설정 등은 `SecurityConfig`·`application.yaml` 을 따른다.

### 6.2 데이터·동시성

- **MySQL** + JPA. 테스트 프로파일은 **H2** 등으로 대체 가능.
- **Redis**: 캐시, 최근 검색어, 조회수 누적, **Redisson** 기반 분산 락 등.
- **P6Spy** 등으로 SQL 추적(프로파일 선택).

### 6.3 외부 연동

- **AWS S3**(또는 로컬/MinIO) 파일 저장.
- **국세청(NTS)** 사업자번호 검증 클라이언트.
- **SMTP** 메일 발송, **Thymeleaf** HTML 메일 템플릿(`spring.thymeleaf.enabled=false` 로 웹 MVC는 비활성).

### 6.4 관측·문서

- **Springdoc OpenAPI** 로 API 문서화(`swagger` 프로파일).
- **JaCoCo**·**SonarQube** Gradle 설정으로 커버리지·정적 분석 연동 가능.

### 6.5 명시적 비포함·미사용

- **Apache Kafka / Spring Kafka** 는 `build.gradle` 기준 **미사용**.
- **대기열 토큰·영화 좌석·지갑·결제 게이트** 등은 **본 저장소 요구사항에 해당하지 않는다**.

---

## 7. 테스트·품질 요구사항

- 단위·통합 테스트는 **JUnit 5** (`spring-boot-starter-test`), `test` 태스크에 **`spring.profiles.active=test`** 지정.
- **embedded-redis** 로 Redis 관련 테스트를 지원한다.
- **JaCoCo** 리포트 생성 후 SonarQube에서 커버리지 XML을 읽을 수 있도록 구성되어 있다.
- 동시성·예약·Redis 등은 **도메인별 테스트 작성**을 권장하나, 구체 시나리오는 팀 규칙에 따른다.

## 8. 문서·구현 싱크 유지

- API 세부 URL·요청 본문은 **Swagger UI** 및 각 `I*Controller` 주석이 최우선이다.
- 패키지·기술 스택 요약은 `docs/PROJECT_STRUCTURE.md` 를 참고한다.
